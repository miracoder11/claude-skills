# 教学文档生成器

## 作用

为任何开源项目自动生成高质量的教学文档，帮助学习者从零开始理解项目原理。

## 核心方法论

### 1. 认知科学原则 - 分层理解

**三层递进**：

```
README.md (抽象层) ← 理解"为什么这样设计"
    ↓
minimal.py (骨架层) ← 抓住"最核心的概念"
    ↓
one_pager.py (细节层) ← 学习"具体怎么实现"
    ↓
源代码 (参考层) ← 完整的生产代码
```

**判断标准**：
- 如果只能看懂源码 → 说明没有抽象层文档
- 如果能看懂 minimal.py → 说明掌握了核心概念
- 如果能看懂 one_pager.py → 说明理解了实现细节
- 如果能自己写出来 → 说明真正掌握了

### 2. 场景化教学 - 从具体到抽象

**不要**：直接定义概念
```markdown
"什么是自动微分？自动微分是一种计算梯度的方法..."
```

**要**：从具体场景切入
```markdown
"想象你在训练一个神经网络识别手写数字...你输出'0.3'但应该是'1.0'。
现在的问题是：你应该如何调整成千上万个参数？
这就是深度学习的核心问题...micrograd 的思路是："
```

### 3. 冗余清晰 - 多角度解释

**重复不是坏事**，当讲到重要概念时：

第一次：用类比解释（银行存款记录）
第二次：用代码示例（`a.grad += out.grad`）
第三次：用数学公式（∂c/∂a = 1）
第四次：用可视化（计算图）

每次重复都是帮助不同学习风格的人理解。

### 4. 框架大图 - 整体认知

每个教程必须包含架构图（ASCII Art）：

```
┌─────────────────────────────────────┐
│              用户代码                │
├─────────────────────────────────────┤
│            神经网络层 (nn)            │
├─────────────────────────────────────┤
│            核心层 (engine)           │
└─────────────────────────────────────┘
```

**作用**：
- 让读者在阅读细节前，先有整体认知
- 了解各模块之间的数据流和控制流
- 便于后续查阅（"这个模块在整体中的位置是..."）

### 5. 大量实例 - 冗余场景强化

**核心原则**：对于新、困难、重要的概念，必须用大量不同场景和实例反复强化。

**判断哪些概念需要大量实例**：
- ✅ 新概念：读者第一次接触的技术
- ✅ 困难概念：抽象、反直觉的机制
- ✅ 重要概念：整个项目的基础，后续依赖它

**实例类型要多样**：

1. **生活场景类比**（至少 2-3 个）
   - 银行系统、物流系统、工厂流水线...
   - 不同类比帮助不同背景的人建立联系

2. **代码场景实例**（至少 3-5 个）
   - 从简单到复杂：单变量 → 多变量 → 嵌套结构
   - 从正常到异常：常见用法 → 边界情况 → 错误处理

3. **对比场景实例**（至少 2-3 个）
   - "如果有会怎样" vs "如果没有会怎样"
   - "用传统方法" vs "用这个方法"

4. **逐步演化实例**（至少 1 个）
   - 从最简单版本开始
   - 逐步添加复杂度
   - 每一步解释"为什么需要这样改"

**示例 - 解释"KV Cache"概念**：

```
【生活类比 1】
想象你做数学题，每次都要重新计算 1+1=2，2+2=4...
如果把这些结果记在纸上，下次直接用，就快多了。KV Cache 就是这个"草稿纸"。

【生活类比 2】
像餐厅的点菜单。第一次你点"宫保鸡丁"，服务员记下来。
后续你再加"多辣"时，服务员直接在原单上改，而不是重新问一遍你点了什么。

【代码实例 1 - 无 Cache】
for i in range(10):
    output = model(input)  # 每次都重新计算，慢

【代码实例 2 - 有 Cache】
cache = {}
for i in range(10):
    if i not in cache:
        cache[i] = model(input[i])
    output = cache[i]  # 直接用，快

【代码实例 3 - 对比】
# 无 Cache：100 tokens 需要 100 × 100 = 10,000 次计算
# 有 Cache：100 tokens 只需 100 × (100+1)/2 = 5,050 次计算
# 速度提升约 2 倍！

【演化实例】
版本 1：每次从头计算（原始方法）
版本 2：缓存上一层的 key/value（初步优化）
版本 3：增量更新缓存（最终优化）
```

### 6. 预测误差驱动 - 主动制造认知冲突

**核心原则**：利用"预期 vs 实际"的差距最大化学习信号。

**神经科学基础**：
- 当预测误差出现时，多巴胺系统释放学习信号，促使大脑更新图式
- 2023年《Neuron》研究：预测误差时前额叶-顶叶 theta 波功率显著增加
- 实证效果：主动制造适度预测误差的学习者，新问题表现高 35%

**在教程中的应用**：

**方法 1：预期-实际对比表**

```markdown
### 预期 vs 实际

| 维度 | 你的直觉/预期 | 实际实现 | 差距说明 |
|------|--------------|---------|---------|
| 反向传播计算 | 从后往前逐层计算 | 拓扑排序后计算 | 为什么不直接反向？因为可能有循环依赖 |
| 梯度更新方式 | 梯度 = 新值 | 梯度 += 新值 | 为什么用 +=？因为多个输出可能影响同一输入 |
```

**方法 2：认知冲突练习**

在每个模块后加入"反直觉问题"：

```markdown
### 反直觉挑战
问题：如果把 `a.grad += out.grad` 改成 `a.grad = out.grad` 会怎样？

[先思考 1 分钟...]

答案：当多个输出都依赖同一个输入时（如 `c = a + b; d = a * b`），
用 `=` 会导致后面的梯度覆盖前面的，最终梯度值错误。
这就是为什么需要 **累加** 而不是 **赋值**。
```

**方法 3：预测-验证循环**

在代码示例中加入"预测阶段"：

```python
# 预测：这行代码会输出什么？
result = tensor(2.0) + tensor(3.0)
print(result.data)  # 你的预测：_____

# 验证
print(result.data)  # 实际输出：5.0

# 解释
如果预测正确：恭喜，你理解了标量加法
如果预测错误：检查你是否忽略了 backward() 的调用时机
```

**实施检查清单**：
- [ ] 每个模块是否至少有 1 个"预期 vs 实际"对比？
- [ ] 是否有反直觉问题挑战读者的直觉？
- [ ] 是否有预测-验证代码示例？

### 7. 交错对比 - 横向连接深化

**核心原则**：通过项目间对比，理解设计权衡，建立知识网络。

**对比维度**：

1. **同一问题，不同方案**
   - Attention 计算：FlashAttention vs 标准 Attention
   - Agent 工具调用：Function Calling vs ReAct 框架

2. **同一方案，不同实现**
   - nanoGPT vs nanochat：同样基于 Transformer，一个用于训练，一个用于推理
   - autogen vs langgraph：同样多 Agent，一个侧重对话，一个侧重工作流

3. **能力边界对比**
   - "这个框架能做什么，不能做什么？"
   - "什么场景应该用 A，什么场景应该用 B？"

**对比表格模板**：

```markdown
| 维度 | 方案 A | 方案 B | 推荐 |
|------|--------|--------|------|
| 适用场景 | 小规模 | 大规模 | 根据数据量 |
| 性能 | O(n²) | O(n) | 大数据用 B |
| 复杂度 | 简单 | 复杂 | 快速原型用 A |
```

**潜在方案思考**：

```markdown
## 延伸思考

### 当前方案的优势
- ...

### 当前方案的局限
- 局限 1：[具体说明]
- 局限 2：[具体说明]

### 潜在改进方向
- 方向 1：[具体说明 + 简要原理]
- 方向 2：[具体说明 + 简要原理]
- 方向 3：[具体说明 + 简要原理]

### 相关项目对比
- 项目 X：[相似点] + [不同点]
- 项目 Y：[相似点] + [不同点]
```

**示例 - Agent 框架对比**：

```
【问题】如何让多个 Agent 协作？

【方案 1：AutoGen - 对话驱动】
- 思路：让 Agent 自由对话，自然形成协作
- 优势：灵活、适合开放式任务
- 劣势：难以控制、可能无限循环

【方案 2：LangGraph - 工作流驱动】
- 思路：预先定义状态转换图，按图执行
- 优势：可控、可预测、易调试
- 劣势：不够灵活、需要预设计

【方案 3：NanoBot - 事件驱动】
- 思路：通过消息总线异步传递事件
- 优势：解耦、易扩展、支持并行
- 劣势：调试困难、需要理解异步

【选择建议】
- 创意探索任务 → AutoGen
- 生产业务流程 → LangGraph
- 高并发系统 → NanoBot
```

## 输出格式要求

### README.md 结构

```markdown
# 项目名教程

## 0. 背景与动机
### 0.1 从一个具体场景说起
[用生动的场景引入问题]

### 0.2 为什么需要它？
[对比传统方法的痛点]

### 0.3 设计思路
[核心洞察 + 设计决策表]

## 1. 框架大图
[ASCII 架构图 + 数据流说明]

## 2-N. 模块详解
每个模块包含：
### 抽象理解
[生活类比 + 本质作用]

### 实现细节
[关键代码 + 设计决策 + 数学公式]

## X. 一页代码
[参见 one_pager.py]

## Y. 超精简版
[参见 minimal.py]

## Z. 面试要点
[常见问题 + 答案要点 + 代码审查 checklist]

## W. 延伸思考
### 局限性分析
[当前方案的边界和不足]

### 改进方向
[潜在优化方案 + 简要说明]

### 项目对比
[与其他项目的对比表格：相同点、不同点、适用场景]
```

### one_pager.py 要求

**详细注释**：
- 每个类：说明作用和类比
- 每个方法：说明参数形状（B, T, C 等）、计算公式、为什么这样设计
- 每个关键行：注释"为什么这样做"

**可运行**：代码必须能直接运行（或说明依赖）

### minimal.py 要求

**目标**：用最少的代码概括**整个项目**的核心架构和流程

**关键原则**：
1. **必须覆盖项目的所有核心组件**，不是只选一个
   - 例如 nanoclaw：Channel + Queue + Container + Skills + Orchestrator
   - 例如 micrograd：Value + backward + 拓扑排序

2. **可以使用伪代码**：
   - 对于复杂组件，用注释描述逻辑而非完整实现
   - 重点展示**数据流和控制流**
   - 标注"伪代码：实际实现会..."

3. **行数范围**：
   - 简单项目（单一核心概念）：50-100 行
   - 中等项目（2-3 个核心组件）：100-200 行
   - 复杂项目（4+ 个核心组件）：200-400 行
   - **目标**：每个核心组件至少 30-50 行

4. **架构图优先**：
   - 开头必须有 ASCII 架构图
   - 展示组件之间的关系和数据流向

**对比 one_pager.py**：
| 维度 | one_pager.py | minimal.py |
|------|-------------|------------|
| 目标 | 可运行的实现 | 架构理解 |
| 抽象度 | 具体实现 | 高层伪代码 |
| 覆盖度 | 深度（细节） | 广度（全貌） |
| 可运行性 | ✅ 必须可运行 | ❌ 不需要 |

**常见错误**：
- ❌ 只展示一个组件（如只写 SkillEngine）
- ❌ 试图让所有代码都可运行
- ❌ 忽略项目的主要部分
- ✅ 用伪代码展示完整架构
- ✅ 注重数据流而非实现细节

## 禁止事项

### 基础结构
- ❌ 不要用抽象定义开场，要从具体场景切入
- ❌ 不要省略框架大图
- ❌ 不要只给代码不给解释

### 代码质量
- ❌ 不要让 one_pager.py 超过 300 行（太长就精简）
- ❌ 不要让 minimal.py 和 one_pager.py 差距太小

### 大量实例
- ❌ 新/难/重要概念不要少于 3 个实例
- ❌ 不要只给一种类型的实例（生活类比/代码/对比/演化都要有）
- ❌ 不要让所有实例难度相同（要有梯度：简单 → 复杂 → 边界）

### 预测误差驱动
- ❌ 不要只给正确答案，要先让读者预测
- ❌ 不要回避反直觉现象，要主动指出并解释
- ❌ 不要只说"这样做"，要解释"为什么直觉错了"
- ❌ 预测-验证示例必须给足思考空间（不要立即公布答案）

### 交错对比
- ❌ 不要只说"这个方案好"，要说出"什么时候好，什么时候不好"
- ❌ 不要只列对比表格，要给出选择建议
- ❌ 延伸思考不要只说"可以优化"，要列出具体的优化方向
